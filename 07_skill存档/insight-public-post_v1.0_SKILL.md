# 知识库 insight → 公开版对外帖（B 站 AI 开发者小站）

把一篇知识库内核档（含双链、面向自己的复盘）改写成**任何陌生人都能冷读**的公开帖。分发链路：内核档 → 公开版（本 skill 产出）→ 用户手动发 B 站小站。**发帖动作永远由用户手动完成**，本 skill 交付到「可整篇粘贴的成稿」为止。

## 第 0 步：定位内核档

- 默认取**本次会话刚沉淀/刚讨论的那篇**内核档；没有明确对象时问用户要路径。
- 内核档还没写时，先完成内核沉淀（进知识库对应栏目 + 索引/MOC 登记），再回来做公开版——顺序不能反，公开版的每个事实都要能溯源到内核档。
- 动笔前先读两份参照：
  1. 内核档全文（事实来源，公开版不得新增内核档里没有的事实）；
  2. 黄金样例 `D:\AIGC工作站\知识库\08_对外分发\AI日报全绿却停更_云端定时任务连环翻车复盘_公开版.md`（语气、结构、脱敏尺度的基准）。

## 第 1 步：写公开版

### 体例硬规则（08_对外分发的既有规范）

- **独立成篇**：正文零 `[[双链]]`、零库内引用、零「见另一篇」，陌生人冷读无障碍；
- **名词小抄**：正文用到的术语（CI、workflow、runner、OIDC 之类），在首次出现附近用引用块集中白话解释——读者是「所有用户」，不全是开发者；
- **全角标点**：正文中文标点一律全角（，。：），这是社交平台文案的呼吸感规则，别照抄内核档的半角逗号；
- **文件命名**：`{口语化标题}_公开版.md`，存入 `D:\AIGC工作站\知识库\08_对外分发\`。

### 语气（B 站人设：技术向 + 整活自黑）

- 短句，皮，不要官方腔：禁止「本文将介绍」「综上所述」这类开场收场；
- 叙事优先于说教：有事故就从事故现场讲起（"打开一看：绿的，全是绿的"），钩子放在方法论前面；
- 自黑是好素材：自己被自己写的代码骗了、白烧了 API 钱，这类桥段照实写，比装专家可信；
- 结尾给一段**可以直接抄走的清单**（心得/口诀/步骤），最后一句收个皮的。

### 内容改写要点

- **只做翻译不做创作**：把内核档的事实翻译成白话，可以删细节、不可以编事实；内核档里标注「未验证/待观察」的结论，公开版同样要留诚实边界或干脆不写；
- **叙事数字零虚构**：凡是描述「这次发生了什么」的数字（耗时、次数、金额、轮数）都必须能在内核档里指认出处——读者会把「排查了三小时」当真事传播，被扒就是公信力事故；没有出处就用定性表达（「折腾了半天」可以，「3 小时」不行）。建议句里的惯用语（「花一分钟自查」「十秒就能验」）不算叙事数字，可以用，但别密集堆。写完在交付说明里列出全部叙事数字及各自出处——用可验的清单，别写「未虚构任何数据」这类不可验的绝对声明；
- **名词小抄以「跟不上主线」为线**：判断标准不是「所有术语都解释」，而是「读者不懂这个词就无法理解本文主线」——这类词必须解释（实测最容易漏的恰是 iframe、CI 这种「感觉大家都懂」的词），修饰性长尾术语（懒加载、节流之类顺带提到的）可以不解释。写完做一次回扫：把正文首现术语列成清单逐个判断该不该解释，凭印象扫会稳定漏词；
- **脱敏**（发布前逐项过一遍）：
  - 匿名化供应商与账户：中转站/代理商写成「API 中转站」，模型名视内容敏感度决定是否点名（默认匿名成「贵的/便宜的」，用户可要求还原）；
  - 抹掉能定位到账户或机器的信息：本机路径、服务器 IP、仓库 secret 名、余额截图式精确数字（保留量级即可）；
  - **内核档里的真实标识符不得原样出现**：仓库名/base 路径（如 `/above-the-web`）、域名、项目代号——发布前拿内核档里的专有字符串在成稿里搜一遍，命中的一律换成示例值（`/my-site` 这类）；判断标准是「读者拿这个串能不能搜到你」；
  - 可复用的数据点（成本对比、轮数、耗时）**保留**，这是帖子的传播点；
- **长度**：1500-2500 字为宜，超了就砍过程细节、保住坑和结论。

## 第 2 步：归档、登记、推送

1. 成稿写入 `08_对外分发/`；
2. 在 `08_对外分发索引.md` 的「已分发清单」表格登记一行：`| 文件名（B站AI开发者小站首发） | 对应内核文档路径 | 主题一句话 |`——溯源关系写在索引里，正文保持零双链;
3. 按知识库 CLAUDE.md 惯例**一并 commit 并 push**（提交信息用「分发:…」前缀），push 即触发官网同步；
4. 回复用户：给出成稿文件链接，说明可整篇粘贴发小站，并提示已做的脱敏处理（如模型名已匿名，想点名可还原）。

## 边界

- 不替用户登录/发帖，不操作浏览器；
- 内核档之外的事实缺口（比如用户想补的新数据）→ 问用户，不要脑补；
- 如果内容涉及公司（鲸海拾贝）内部信息，先提醒用户确认可公开再动笔。
