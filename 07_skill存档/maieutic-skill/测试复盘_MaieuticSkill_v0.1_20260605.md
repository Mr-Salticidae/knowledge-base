# 测试复盘 · Maieutic Skill v0.1

> 测试时间：2026-06-05  
> 测试对象：[[maieutic-skill/SKILL.md]]  
> 当前结论：首轮 MVP 测试通过，待真实用户测试继续验证。  
> 核心修正：Insight / Beacon 从固定输出改为事件触发输出。

---

## 一、测试结论

`maieutic-skill v0.1` 已完成首轮五个 MVP 用例测试，覆盖知识解释、学习路径澄清、迷茫反思、AI 公益项目方向澄清、Claude Skill 学习案例信息收集。

当前版本可以作为通用共学 Skill 的首个可用版本。它能根据问题类型切换模式，并在需要时使用 Research-Assisted Mode。首轮测试后最重要的产品结论是：**Insight / Beacon 不是仪式化结尾，而是按真实认知推进和行动需要触发的输出。**

## 三、关键修正

### Insight / Beacon 改为事件触发

原规则倾向于在完整对话后输出 Insight / Beacon。测试发现，这会在知识解释或早期澄清轮中产生仪式化、模板化、空洞化风险。

修正后规则：

- Knowledge Mode 默认只解释概念，不输出 Insight / Beacon。
- Exploration Mode 早期只问关键分流问题，不默认输出 Insight / Beacon。
- Insight 只在出现真实认知转折、关键区分或盲区命名时输出。
- Beacon 只在用户明确需要实操指导、下一步、练习、应用或测试时输出。
- Insight 与 Beacon 互相独立，不因为一个成立就强行输出另一个。

---

## 四、为什么要这样改

Maieutic 的核心不是给每轮对话套一个固定结构，而是帮助使用者把问题想清楚。

固定 Insight / Beacon 会带来三个问题：

1. **仪式化**：用户只问概念解释，却被迫接收行动任务。
2. **模板化**：每轮都像套同一个结尾，降低对话自然度。
3. **空洞化**：没有真实认知转折时生成 Insight，会接近鸡汤。

事件触发规则更符合“问题决定模式”：

- 需要解释时解释。
- 需要澄清时澄清。
- 真的有认知推进时再命名 Insight。
- 真的需要行动时再给 Beacon。

---

## 五、当前可用结论

`maieutic-skill v0.1` 可作为首个可用版本进入后续真实场景测试。当前适合的使用场景：

- 明确知识问题的简洁解释。
- 学习路径不清时的方向分流。
- 迷茫、过载、卡住时的反思澄清。
- 创作 / 产品 / 公益项目方向的结构化讨论。
- 需要案例或资料时的信息收集辅助。

当前不适合扩大为：

- 课程生成器。
- 资料检索机器人。
- Web App / 数据库系统。
- 长期记忆产品。
- 公益垂直子 Skill。
