# Work Weekly Report · 公司工作周报工作流

## 角色定位

你帮跳蛛先生生成**提交给公司的工作周报**。这不是创作复盘，是一份**对外汇报**文档——读者是公司/上级，要看清「这周干了什么、下周干什么、卡在哪、想明白了什么」。

两个本质特征，决定了它和库内其他 skill 不同：

1. **跨仓库聚合**：跳蛛先生的工作散在 E 盘十几个仓库里（公司平台 `taowhale-site`、AIGC 创作、知识库、游戏/桌面应用、主页站、工具…）。周报要把一周横跨所有仓库的活动收拢成 4–6 条工作线。
2. **产物落在另一个仓库**：本 skill 文件在知识库 `07_skill存档/`，但**周报成品写到 work-reports 仓库**（`E:\工作报告`），并推送到它自己的远程，**不进知识库**。

---

## 触发判断

- "写周报"、"本周工作周报"、"这周的周报"、"生成周报交公司"、"提交给公司的周报"；
- 周五/周末/周一节点，要交上一周的工作汇报。

**默认时间盒**：上一个完整自然周（周一 00:00 → 周日 23:59）。作者另指定则按指定。先看 work-reports 已有最新一期覆盖到哪一周，**接着往下排，别重复或跳周**。

## 关键路径与格式约定（照抄，别自创）

| 项 | 约定 |
|---|---|
| 周报输出仓库 | `E:\工作报告`（= GitHub `Mr-Salticidae/work-reports`，SSH，分支 `main`） |
| 周报落位 | `E:\工作报告\2026\周报\` |
| 文件命名 | `YYYY-MM-DD_MM-DD-周报.md`（如 `2026-06-22_06-28-周报.md`，下划线连起止日） |
| 标题行 | `# 周报 · YYYY.MM.DD–MM.DD`（点分日期，连接号是 `–`） |
| 取数源·仓库清单 | `E:\GitHub仓库对应关系.md`（列了哪些是本人仓库、哪些是外部上游） |
| 月报 / 项目材料 | 月报落 `2026\月报\`；项目过程材料落对应月份项目文件夹（见 work-reports README） |

固定三段式（缺一不可）：**一、本周工作总结** ／ **二、下周工作计划** ／ **三、协助 · 思考 · 总结 · 成长**。

## 四阶段流程

### 阶段 A · 取数（跨仓库扫描，别靠回忆）

一周横跨十几个仓库，记不全。客观账本是各仓库的 git log。

1. **发现仓库**（不要只读映射文档——它会滞后，如 `fraud-desk`/`zombie-world` 一度未登记）。扫 E 盘工作区里所有 git 仓库：
   ```bash
   for d in "E:/"*/ "E:/AIGC工作站/"*/ "E:/工具/"*/; do
     [ -d "$d.git" ] && echo "${d%/}"
   done
   ```
   再对照 `E:\GitHub仓库对应关系.md` **剔除外部上游**（`remotion`、`video-use` 等只读跟踪的非本人仓库，不计入工作量）。

2. **逐仓库抓本周提交**：
   ```bash
   SINCE="YYYY-MM-DD 00:00"; UNTIL="YYYY-MM-DD 00:00"   # UNTIL 取周日的次日 00:00
   git -C "$REPO" log --since="$SINCE" --until="$UNTIL" \
     --pretty=format:"%ad | %s" --date=format:"%m-%d"
   git -C "$REPO" rev-list --count --since="$SINCE" --until="$UNTIL" HEAD   # 该仓提交数
   ```
   提交信息前缀即分类信号：`新档/沉淀/复盘/改版/新工具/新工程/精简/修复/发布`。

3. **补 git 抓不到的**：很多创作产出（AIGC 短片/图集立项与发布、平台分发与数据、线下沟通、服务器侧操作）不在 git 里。**列待补清单问作者**：本周发布了什么、平台数据（CTR / 点赞 / 播放 / 涨粉 / 投流）、有无新立项/交付、外部凭证阻塞是否解除。问不到的数据如实标，别编。

### 阶段 B · 归集到工作线

把所有仓库的提交，按**稳定的工作线**归类（顺序可调，但公司平台要可见）：

| 工作线 | 典型来源仓库 |
|---|---|
| **公司平台** | `taowhale-site`（公司官网/知识库模块/安全加固）、`xiaoe-learning`（小鹅通数据采集） |
| **个人创作主业**（AIGC 短片/图集/封面） | `AIGC工作站`(aigc-creative-archive)、`bilibili-ctr-pipeline`、`fraud-desk`、`目标是成为 Prompt 大师` |
| **知识库沉淀**（含新 Skill / 首发方法论） | `knowledge-base` |
| **游戏与桌面应用** | `zombie-world`、`last-stand`、`desk-pond`、`livelink` |
| **工作区与维护** | `above-the-web`(主页站)、`工作报告`、`工具/*`、仓库映射维护 |

> 工作线随业务演变，以**当周实际有产出的**为准，没动的线不硬凑。

### 阶段 C · 写周报（三段式，照真实格式）

#### 一、本周工作总结

- 按工作线分组，每组 `**N. 工作线名（括注：本周状态，如「承上周计划，本周收官交付」）**`，下面跟 `-` bullet 写具体动作与结果。
- **加粗关键术语 / 量级 / 里程碑**（如 **100 母版 ×（BASE+9 变体）**、**FTS5 trigram 检索**、**v1.0→v1.2**）。
- 写**推进与结果**，不是流水账（"完成最终交付：X" 优于 "做了 X"）。
- 用项目代号（`47_众`、`46_含泪凝视`、`EP01 SCARLATTA`）保持与内部一致。

#### 二、下周工作计划

- `-` bullet，每条 = **项目名 + 具体可执行的下一步**（"出六/五张首帧 → 静态占位粗剪卡节奏"，不是"继续推进"）。
- 优先排：承本周未收口的、公司平台、有 deadline 的。

#### 三、协助 · 思考 · 总结 · 成长

四个子块，**标题加粗**：

- **需协助**：阻塞项，尤其外部凭证/额度。**复发的标「沿用上周/前两周阻塞，未解除」**——让公司看到卡点是外部依赖、已持续多久。
- **思考**：2–3 条原则级提炼，`**一句话原则**：展开`。诚实、有洞察（范例：「诚实是最便宜的护栏」「机制认知 > 参数调校」「经验要资产化、可交接」）。
- **总结**：一段话，本周主线叙事——哪条线是重头、从什么切到什么、双线/三线如何并进、显性阻塞是什么。
- **学习成长**：2–3 条本周新掌握的能力/认知（"掌握 OCR 驱动的量产质检闭环""打通 SQLite FTS5 中文全文检索"）。

### 阶段 D · 归档与提交（写到 work-reports，不进知识库）

1. 写入 `E:\工作报告\2026\周报\YYYY-MM-DD_MM-DD-周报.md`。
2. **在 work-reports 仓库**提交并推送：
   ```bash
   git -C "E:/工作报告" add "2026/周报/<文件名>"
   git -C "E:/工作报告" commit -m "周报:YYYY.MM.DD–MM.DD"
   git -C "E:/工作报告" push origin main
   ```
3. **不要**把周报写进知识库、不要在知识库里 commit 它。两个仓库分开。
4. 若本周维护动作里包含「work-reports 纳入版本管理 / 归档上周周报」，可在本周总结的「维护」线提一句（真实周报就这么记的）。

## 跨周连续性（每期必做）

1. **回填上周计划**：打开 work-reports 上一期的「二、下周工作计划」，本周总结里体现哪些收口了（"承上周计划，本周收官交付"），下周计划里顺延哪些。
2. **阻塞续期**：外部凭证/额度类阻塞，在「需协助」标明已沿用几周——这是周报对公司最有用的信号之一（暴露外部依赖卡点）。
3. **主线漂移**：总结里点出本周主线落在哪（如「从短片创作切到产品化 + 内容站运营」）。

---

## 写作护栏

- ✅ **诚实优先**：夸大/误导要改回如实（真实案例：「自封神图」改回《空》、「扣1领源码」导流话术全系去除）。周报是长期信任资产。
- ✅ **加粗关键信息**：术语、量级、里程碑、版本号加粗，方便快速扫读。
- ✅ **公司平台可见**：个人创作占比再大，公司平台线（taowhale-site 等）要清晰呈现，别被淹没。
- ✅ **数据带来源**：CTR/点赞/播放有就写、标来源；没有就标「数据待回收/待补」。
- ❌ 不编数据、不写"大概/应该"凑客观数字。
- ❌ 不用流水账（"改了几个文件"），写推进了什么。

## 与相邻 skill 的区别

| Skill | 做什么 | 产物去哪 |
|---|---|---|
| `aigc-postmortem` | 单个作品/比赛纵向复盘（事实先行） | 知识库 |
| `knowledge-base-curator` | 一次答疑成果沉淀 + 对外学员版 | 知识库 / `08_对外分发` |
| **`work-weekly-report`（本）** | **跨工作区所有仓库**的一周活动 → 公司三段式周报 | **work-reports 仓库** |

本 skill 的「知识库沉淀」工作线，可引用 `knowledge-base` 当周的复盘/新 Skill 作为素材；但它是**对公司的汇报**，不是知识库内部资产，归 work-reports。

## 禁止行为

- ❌ 把周报写进知识库 / 在知识库 commit（应落 `E:\工作报告` 并推 work-reports 远程）
- ❌ 只读映射文档取数（会漏掉未登记的新仓库；必须扫盘发现 git 仓库）
- ❌ 把外部上游仓库（remotion / video-use）的提交计入工作量
- ❌ 跳周或与已有最新一期重叠（先看 work-reports 排到哪周）
- ❌ 丢掉三段式任一段，或漏掉「需协助/思考/总结/学习成长」子块
- ❌ 编平台数据 / 把流水账当工作总结
- ❌ 复发阻塞不标「沿用上周」（公司看不到外部依赖卡了多久）
