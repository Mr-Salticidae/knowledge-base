# 我的 AI 日报「全绿」了好几天，其实一篇都没发出去：一次连环翻车复盘

> 适用：你想用 GitHub Actions + AI（Claude / GPT 都一样）做「每天自动干点活」的流水线——自动写日报、自动更新数据、自动发内容。我踩的这 6 个坑，大概率你也会踩。

## 先交代背景

我的个人网站上有个「AIGC 快讯」板块：每天早上 9:30，云端定时任务自动叫醒 Claude，让它搜过去 24 小时的 AI 新闻、写成一期日报、构建、提交、上站。全程无人值守，听起来很美。

然后有天早上我打开网站：停更了。

打开 GitHub Actions 一看：**绿的，全是绿的**。每一次运行都写着 success。

排查完我才发现真相，比停更本身刺激多了——**这条流水线从上线起，就没有真正成功跑通过哪怕一次**。之前的每一期都是我手动发的，而「全绿」是它精心伪装出来的。

下面是连环 6 坑，按翻车顺序上菜。

> 名词小抄：
> - **GitHub Actions**：GitHub 的自动化流水线，可以定时跑脚本。
> - **workflow**：一条流水线的配置文件（.yml）。
> - **schedule / cron**：定时触发器，比如「每天 9:30」。
> - **runner**：GitHub 临时租给你跑任务的那台云端机器。

## 坑一：定时器压根没响过

cron 写得没问题，workflow 状态也是启用的，但 GitHub 从来没有在 9:30 触发过它——一次都没有。

这是 GitHub 的老毛病：**新建的 schedule 首次触发不可靠**，高峰期可能延迟甚至直接丢弃。同仓库里另一条老流水线的定时任务一直好好的，就这条新的装死。

对策：别信「配好了就会跑」。上线后手动触发一次验证全流程，然后**盯着它第一次自然触发**，没等到就换个冷门分钟（比如 9:47，别用整点半点）。

## 坑二 & 坑三：AI 步骤的两道门禁

手动触发后，Claude 那一步直接报错。连吃两个：

1. 缺 `id-token: write` 权限——claude-code-action 要用 OIDC 换 token，workflow 的 permissions 里不写这行就是死。
2. 仓库没装 Claude GitHub App——不想装的话，给 action 显式传一个 `github_token: ${{ github.token }}` 就能绕开。

这两个都是文档角落里的硬性要求，报错信息倒是很诚实，照着加就行。

## 坑四：白名单把 AI 的手脚全捆了

修完门禁再跑：Claude 干了 8 分钟，一个字没写出来。

看日志：**16 次工具权限被拒**。我只给它开了 `curl` 的白名单，但它搜新闻必然要 `curl xxx | grep yyy` 这种管道组合——而组合命令要求**每一段**都在白名单里。于是它每次伸手都被打，换个姿势再伸手，再被打，8 分钟后心灰意冷空手收工。

对策：白名单要么放够（把 grep/sed/jq/head 这些无害的只读工具都加上），要么在提示词里明说哪些路走不通。**别让 AI 自己去撞墙，它撞墙的每一下都是你的 token 钱。**

## 坑五：最冤的一坑——稿子写好了，丢了

白名单放开后，Claude 真的搜到了新闻、写好了当天的日报、本地也提交了。就在最后 push 的前一步，翻车：

```
error: cannot pull with rebase: You have unstaged changes.
```

原因：提交前有一步 `npm run build` 自检，而构建脚本会顺手改写仓库里的一些静态文件（加缓存参数）。工作区一脏，rebase 拒绝执行，push 没发生。**runner 是临时机器，任务一结束就销毁——那期写好的日报跟着一起火化了。**

对策：在 pull/push 之前把构建副产清掉（`git checkout -- .`）。凡是「构建之后还要做 git 操作」的流水线，都埋着这颗雷。

## 坑六：真正的报错，藏在你看不见的地方

以上全修完，又失败了一次，而且这次日志里什么线索都没有，只有一句干巴巴的 `is_error: true`。

claude-code-action 的完整执行记录（包括 API 报错原文）写在 runner 本地的一个 JSON 文件里，**不会出现在 workflow 日志里**。我给流水线加了一步「失败时把这个 JSON 的末尾打印出来」，一加，10 秒定位：

```
API Error: 402 Insufficient credits. Current balance: $-0.051428
```

……API 中转站余额是负的。之前那次 8 分钟的空手运行把钱烧穿了。

对策：接入任何黑盒 action，**第一个 commit 就加诊断步骤**，把它的执行输出暴露到日志。先能看见，再动手修。

## 充值之后，全线贯通

第二天早上，这条流水线第一次真正无人值守跑通：搜稿、写 JSON、构建自检、提交、部署，一气呵成。看着那期不是我发的日报出现在网站上，还挺感动的。

## 彩蛋坑：换便宜模型，账单反而翻倍

跑通之后我想省钱，把模型从旗舰款换成了单价只有 1/3 的型号。结果：

| 模型 | 单价 | 轮数 | 实际花费 |
|---|---|---|---|
| 贵的 | 1x | 约 40 轮 | $4.04 |
| 便宜的 | 约 1/3 | 178 轮 | **$7.26** |

便宜的模型干活更「勤奋」：RSS 搜完不满足，跑去爬搜索引擎结果页、写 for 循环批量查询，轮数翻了 4 倍多，把单价优势整个吃掉还倒贴。

对策：**成本 = 单价 × 用量**，agent 型任务换模型必须实测总账单。同时给 AI 设硬上限（`--max-turns`）和明确的收工条件（「凑够 5-8 条就立刻写文件交差，别追求完美」）。AI 不知道你的钱包上限，你得告诉它。

## 四条心得（可以直接抄走）

1. **全绿 ≠ 已发。** 带「兜底不发」分支的流水线，失败会伪装成成功。验收和巡检要盯业务产物（提交记录、线上页面），别看 CI 徽章的脸色。
2. **手动跑通会掩盖自动路径。** 手动发过一次，幂等检查就会让自动路径直接跳过——你以为验证过了，其实它从没执行过。验证要分三层分开记账：人工跑通 / 自动路径真实执行一次 / 定时器自然触发。
3. **黑盒 action 先接可观测性。** 报错看不见的时候，修什么都是盲修。
4. **换便宜模型 ≠ 更省。** 单价 × 用量才是账单，给 agent 设停止条件比换模型更省钱。

---

这 6 个坑，每一个单看都不致命，串在一起就是「全绿的停更」。希望你的定时任务第一次就响。
