# 让 AI 助理一句话发飞书文档：一条零依赖 CLI 链路是怎么搭起来的

> 适用：你经常写 Markdown（复盘、报告、教程），要发到飞书给团队或同好看，每次都要手动新建文档、复制粘贴、修排版。这篇讲我们的做法：对 AI 助理说一句"发布到飞书"，几十秒后拿回一条可以直接粘到群里的文档链接——排版完好、权限开好、自己可编辑。整条链路一个 npm 包都不用装。

## 先看效果

```
我：把这篇拆解发布到飞书

AI：1/5 上传 Markdown ...
    2/5 创建导入任务 ...
    3/5 等待转换 ...
    4/5 开启组织内可阅读 ...
    5/5 授权文档给你的账号(可编辑/可管理) ...
    完成: https://xxxx.feishu.cn/docx/QxNzdX……
```

表格、加粗、代码块全部保留，打开就是一篇排好版的飞书云文档。

**名词小抄**

- **CLI**：命令行工具。没有界面，一行命令跑完一件事，最适合交给 AI 助理代劳。
- **飞书开放平台**：飞书给开发者的后台（open.feishu.cn），可以创建"自建应用"。
- **自建应用**：你在自己组织里注册的一个机器人身份。它有自己的 ID 和密钥，能替你调飞书接口。我们给自己的起了个名字叫"小桁"。
- **Markdown**：用纯文本写格式的写作格式，`# 标题`、`**加粗**` 这种。AI 助理的默认输出格式。

## 链路总览：三个角色

1. **一个飞书自建应用**——机器人身份，负责"有权限做事"。只需注册一次，ID 和密钥永远不变。
2. **一个约 260 行的 Node 脚本**——负责"把事做完"。零依赖（不用 npm install，Node 18 以上自带的能力就够），核心就是按顺序调 5 个飞书开放接口。
3. **AI 助理（Claude / Codex 都行）**——负责"听懂人话 + 跑命令 + 出错时排障"。你只说"发布到飞书"，找文件、检查环境、跑脚本、把链接交回来都是它的事。

关键理解：**脚本解决的是"能发"，AI 助理解决的是"一句话就发"。** 两者缺一，这条链路都不成立——没脚本，AI 每次要从零摸接口；没 AI，你每次要自己记命令、修环境。

## 搭建步骤

### 第 1 步：注册自建应用（一次性，10 分钟）

1. 打开 open.feishu.cn → 开发者后台 → 创建自建应用；
2. 在「权限管理」开通**云文档**权限（一档叫 `drive:drive`，覆盖上传、导入、开分享、加协作者），改完**发布一个应用版本**（权限开了不发版不生效）；
   - ⚠️ 网上很多教程叫你再开「通过手机号获取用户 ID」用来自动授权——**别信这条**。我们实测：这条权限只对"用户身份"生效，脚本用的是"应用身份"，用手机号一律报 `99991672`，发版也没用；按邮箱加协作者又报 `1063001`。真正稳的授权办法见下面第 4 步；
4. 在「凭证与基础信息」抄下 **App ID** 和 **App Secret**。

### 第 2 步：脚本要做的五件事

不贴全文（260 行），贴骨架。会用 AI 的直接把这段喂给它，让它照着写就行：

```
1. 拿令牌   POST /auth/v3/tenant_access_token/internal   (用 App ID + Secret 换)
2. 传文件   POST /drive/v1/medias/upload_all
            parent_type 填 ccm_import_open
            ⚠ 必须带 extra 参数 {"obj_type":"docx","file_extension":"md"}
              不带就报 1061004,错误信息完全看不出原因
3. 转文档   POST /drive/v1/import_tasks   (md → 飞书云文档,拿到 ticket)
4. 等转换   GET  /drive/v1/import_tasks/{ticket}  轮询
            job_status: 0=成功 1=处理中 2=排队
5. 开权限   PATCH /drive/v1/permissions/{doc}/public   开「组织内可阅读」
            POST  /drive/v1/permissions/{doc}/members  把你的账号加为协作者
```

凭证不要写死在脚本里。放用户目录下一个小配置文件（比如 `~/.feishu/config.json`），脚本启动时读：

```json
{
  "app_id": "cli_……",
  "app_secret": "……",
  "tenant_domain": "xxxx.feishu.cn",
  "owner_open_id": "ou_……（你本人的 open_id，怎么拿见下方「踩过的坑」第 1 条）"
}
```

### 第 3 步：交给 AI 助理

这一步最容易被忽略，也最值钱：**给脚本写一份 README，把用法、配置位置、常见报错写清楚**。AI 助理读完 README 就会用了，以后你只说人话：

- "把这篇复盘发到飞书" → 它自己找文件、跑命令、回链接；
- 报错了 → 它读错误信息对照 README 的故障速查自己修；
- 换了台机器 → 它发现凭证缺失，会主动向你要 App ID/Secret 重建配置。

更进一步，可以让 AI 把整个流程沉淀成它自己的"技能档案"（Claude 里叫 skill）：下次哪怕换个会话、换台机器，一句"发布到飞书"就能全流程复现，不用重新教。

## 我们踩过的坑（每条都是真事）

1. **自己发的文档自己改不了——而"官方教程式"的解法是错的。** 导入接口生成的文档归"应用"所有，你的人类账号默认只有阅读权。要授权就得把你加为协作者，而加协作者得先知道你的账号 ID（open_id）。教程都教你开"通过手机号获取用户 ID"权限去换——**这条在实测里根本走不通**（应用身份拿不到，报 `99991672`；改用邮箱加协作者报 `1063001`；在文档里搜自己也搜不到）。绕了半天才找到真正稳的一条：**如果你的应用是某个飞书「知识库(wiki)空间」的成员，空间成员表里直接就带着每个人的 open_id，而且读它不需要通讯录权限**。做法：调 `GET /wiki/v2/spaces/<空间ID>/members` 拿到你自己那条的 `member_id`（形如 `ou_xxxx`），填进配置 `owner_open_id`；脚本再用它按 `member_type=openid` 加你为协作者（`full_access`），一次就成。一句话记牢：**授权认 open_id，别认手机号。**
2. **上传接口的暗坑参数。** 不带 `extra` 参数就报 1061004，报错信息里没有任何提示。这种坑只能踩一次记一次——所以要写进 README，让 AI 下次直接绕开。
3. **本地图片不会跟着上去。** 这条链路只传 Markdown 文本，`![](本地图.png)` 导入后是坏链。纯文字加表格的文档完美；带图的要么改成文字描述，要么另走支持传图的重脚本。
4. **凭证会丢，凭据不会变。** 我们有台机器系统盘每次重启都还原，配置文件必丢。后来想通了：丢就丢，App ID/Secret 在开放平台永远查得到，AI 发现配置缺失时问一句、重建一次只要十几秒。**把"凭证丢失"当常态设计，而不是当事故处理。**
5. **发布前先跑连通性测试。** 脚本留一个 `--test` 模式（只验证令牌和权限、不上传），坏凭证在这一步暴露，别带着问题进正式发布流程。

## 可抄清单

- 自建应用只注册一次，权限跟应用走；App ID/Secret 永远可在后台查到；
- 脚本零依赖：Node 18+ 自带 fetch/FormData，够用，别为这点事引入一堆包；
- 五步接口顺序：换令牌 → 传文件（带 extra）→ 建导入任务 → 轮询 → 开分享+授权；
- 授权认 open_id 不认手机号：`owner_open_id` 写进配置（从 wiki 空间成员表拿），否则自己的文档自己只读；
- 凭证放用户目录配置文件，脚本、别的项目、AI 助理共用一份；
- README 写给 AI 看：用法 + 配置位置 + 故障速查，这是"一句话就发"的全部秘密；
- 有条件就让 AI 把流程沉淀成 skill，换机器换会话都能一句话复现。
