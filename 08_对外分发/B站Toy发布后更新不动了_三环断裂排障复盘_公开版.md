# 【踩坑记录】我的 B站 Toy 发布后就再也更新不动了：一次三环断裂的排障复盘

> 适用：你把一个静态网页应用发布到了某个托管平台（B站 Toy、小程序、itch.io 都一样），数据靠 GitHub Actions 定时更新——然后某天发现，**线上的东西不动了**。我这次一口气踩了五个坑，分属三个互不相干的环节，每一个单看都像是“别人的锅”。

## 先交代背景

我做了个台风信息站「风眼」：一个零依赖的静态单页，显示台风实况路径 + 分级应急预案，发布成 B站 Toy（B站的静态页托管，内容跑在它的沙箱里）。

数据链路是经典三段式：

**GitHub Actions 定时抓气象数据 → 提交到仓库、GitHub Pages 分发 → 页面加载时在线拉最新数据。**

上线后我持续迭代了好几版：加了 30 分钟自动刷新、修了数据读取。直到有天对着线上一看——**还是首发那一版，数据停在两天前**。仓库里明明一切正常，CI 绿的，Pages 也是新的。

排查完发现：从发布那天起，**线上就从来没有更新成功过**，而且卡住它的不是一个问题，是三个。

**名词小抄**

- **B站 Toy**：B站的静态网页托管，上传 ZIP 发布，内容实际跑在一个沙箱 iframe 里。
- **GitHub Actions / schedule**：GitHub 的自动化流水线和它的定时触发器（cron）。
- **GitHub Pages**：GitHub 免费静态托管，域名是 `*.github.io`。
- **jsDelivr**：免费 CDN，可以直接镜像 GitHub 仓库里的文件。

## 坑一：CI 校验器把一行合法代码判了死刑

仓库里有条“发布前预检”流水线，第一步跑静态资源校验脚本，两次运行全挂，后面的步骤全部跳过。

看报错，唯一的 ERROR 是：

```
ERROR: index.html: missing local asset: ./ -> .
```

罪魁是页面里再普通不过的一行：`<a href="./">`（品牌 logo 点击回首页）。校验脚本把 `href` 的值一律当“相对路径资产”去解析，`./` 被拼成 `.`，然后在文件清单里找不到名为 `.` 的文件——判死。

**对策：校验器报 ERROR，先复现它的解析逻辑，再动手改。**读一遍校验脚本的源码，确认这是解析盲区而不是代码真有问题，然后选改动成本低的一侧：我把 `href="./"` 改成 `href="index.html"`，一行过检。别上来就大改代码，也别急着骂工具——先搞清楚它到底是怎么判的。

## 坑二：所谓“更新流程”，根本就不存在

预检修好了，我以为可以自动更新了。再一看：仓库里**压根没有任何一条会调用“更新”接口的链路**。更新脚本要求传项目 ID，而配置文件里只记了页面地址，ID 那一栏是空的——当初发布完没人把它记下来。

没有 ID，更新永远无法发生。而这件事 CI 不会报错，因为**从来没有任何东西尝试过更新**。

转机来得很朴素：我打开线上页面查看源代码，B站的外壳页里明晃晃写着：

```html
<script>window.__TOY_META__={toy_id:8980525008896};</script>
```

缺的 ID，平台自己送上门了。更妙的是外壳里的 iframe 地址带着版本后缀：

```
https://…/toy/xxxx/8980525008896-v1/index.html
```

**`-v1`。首发版。铁证如山：发布之后从未更新过，一次都没有。**

**对策（这条我觉得最值钱）：排“线上不更新”这类故障，第一步先抓线上产物的版本指纹**——URL 里的版本后缀、外壳 meta、页脚版本号，一步定性到底是“更新管道坏了”还是“从未走通过”，然后才回头拆管道。顺便，外壳页里往往藏着你缺的元数据。

## 坑三：GitHub 的定时器，整小时整小时地丢班

前两环修好，新版本也过审上线了。第二天早上用户反馈：数据还是一小时前的。

打开 Actions 运行记录，这次看到了很壮观的东西：我的定时任务每小时该跑 4 班，当晚连续几个小时**每小时只被放行约 1 班**，最狠的一段连丢 4 班。而 GitHub 状态页全程绿色 operational——定时任务延迟和丢弃，在官方口径里属于“尽力而为”的正常波动，**没有 SLA**。

加密班次（每小时 8 个触发点）只能缓解，只要触发源只有 GitHub cron，这个风险就消不掉。

**对策：给定时任务配一条 GitHub 之外的兜底触发。**我用的是免费的 cron-job.org，每 15 分钟调一次 GitHub API 强制触发工作流。两个关键设计：

1. **兜底触发的目标不是主任务，而是一个“数据过期了才动手”的守护任务**——GitHub 自己正常时它空转零成本，丢班时它才补抓。幂等，不产生重复提交；
2. Token 用 GitHub 的细粒度 PAT，只授权这一个仓库的 `actions: write`，新员工权限，丢了也炸不了别的。

配置速抄（cron-job.org 里建一个 POST 任务）：

```
URL:  https://api.github.com/repos/<你>/<仓库>/actions/workflows/<守护任务>.yml/dispatches
Body: {"ref":"main"}
Headers:
  Authorization: Bearer <细粒度PAT>
  Accept: application/vnd.github+json
  Content-Type: application/json
```

戏剧性的是：这条兜底上线后的**第一次执行**，就赶上 GitHub 又在丢班（数据已过期 37 分钟），当场补救了一次。上岗第一分钟就开张。

## 坑四：数据源只有 github.io，对大陆用户约等于没有

还有一个更隐蔽的问题：页面在线拉数据只有一个源——GitHub Pages。`*.github.io` 在大陆访问是出了名的不稳，一旦拉取失败，页面静默降级到包内的旧快照。用户看到的就是：**数据永远停在发布那天**。这大概率是“感觉从来不更新”的另一半真相。

**对策：多镜像并行取数，按数据新鲜度取胜。**我改成同时请求 4 个源：

- GitHub Pages（主源）
- `cdn.jsdelivr.net` + `fastly.jsdelivr.net`（同一 CDN 的两个入口，直接镜像仓库文件）
- `raw.githubusercontent.com`（兜底）

要点有两个：一是**比较数据里的时间戳取最新**，而不是先到先得（镜像可能给你缓存的旧数据）；二是 jsDelivr 有缓存 TTL，**数据每次更新后要主动调它的 purge 接口**（`purge.jsdelivr.net/gh/...`），实测 purge 后一分钟内镜像就跟上了。

## 坑五：沙箱 iframe 把电话链接静默吞了

用户还反馈：页面底部的应急电话（110/119/120）点了没反应。本地测试明明好好的。

原因：B站 Toy 的内容跑在一个 `sandbox` 属性齐全的 iframe 里，`tel:` 这类外部协议的页内跳转会被沙箱静默拦截——不报错、不提示，就是没反应。

**对策：检测到自己在 iframe 里时，换三级唤起 + 保底。**依次尝试顶层导航（沙箱通常授予“用户手势下的顶层导航”权限）→ `window.open` → 本框架跳转；同时无论成败，**自动把号码复制进剪贴板并弹提示**——就算三级全被拦，用户也能粘贴拨号。诚实声明：三级唤起在 B站 App 真机上的成功率我还没实测，但“号码已复制”这条保底是稳的。

## 五条心得（可以直接抄走）

1. **版本指纹先定性，再拆管道。**线上产物的版本后缀/meta/页脚版本号，一眼区分“管道坏了”和“从未走通”，少走一半弯路。
2. **校验器 ERROR ≠ 代码有病。**先复现校验器的解析逻辑，分清真缺陷和解析盲区，选便宜的一侧改。
3. **GitHub schedule 没有 SLA，生产级定时任务必须有 GitHub 之外的兜底。**兜底触发“过期才动手”的守护任务，幂等零副作用。（这条我在两个不同仓库以两种姿势验证过了：一个是从来不触发，一个是整点丢班。）
4. **单源在线数据在墙内约等于没有。**多镜像并行、按时间戳取最新、更新后主动 purge 缓存。
5. **平台沙箱会静默吃掉外部协议链接。**`tel:`/`mailto:` 在 iframe 里点了没反应不是 bug 是特性，三级唤起 + 复制保底。

三个环节、五个坑，每一个的表象都是同一句话：“怎么不更新啊”。排障最大的敌人不是坑多，是**所有坑共用一个症状**。

先定性，再定位，一环一环收——希望你的 Toy 第一版之后还有第二版。
