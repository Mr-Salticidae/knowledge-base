## 二、方法论沉淀

### [教程步骤从复盘和 git 历史反推，不从记忆里写]

**核心**：给学员的教程不是回忆录。每一步都要能对应到复盘文档的记录、git 提交或仓库里的真实产物；对不上的步骤不进教程。

**来源**：v0.1 大纲的五步实操、四组前后对比案例，全部从 13 个提交与 `edit/verify/` 截图反推得出，审核零事实性修改。

**操作规则**：
1. 开工先读三样：项目复盘、`git log`、产物目录；
2. 教程里的每个案例先确认素材存在（截图/成片/文档），再写进大纲；
3. 复盘里「没有一次做对的」清单是教程避坑素材的第一来源。

**反例/边界**：项目没有复盘或提交历史散乱时，先补复盘再写教程，否则反推无据。与 [[复盘事实先行原则]] 同根——复盘保真是反推的前提。

### [大纲先行，附「制作侧附注」，两级审核后才写正文]

**核心**：大纲是最便宜的审核对象。让作者和上级在 100 行大纲上做决策，而不是在 2000 字正文上返工；大纲末尾附素材缺口、篇幅预估、删改去向，审核人能一眼评估成本与损失。

**来源**：本次 v0.1→v0.2 大改（结构从 7+1 节改为 4 节+卡片、篇幅减四成）发生在大纲层，正文零浪费。「删改内容去向」附注让作者核对「有没有误删」只花一眼。

**操作规则**：
1. 正文动笔的唯一前提：大纲两级审核全部通过；
2. 大纲每节标注「需要素材」，缺口汇总进文末附注；
3. 版本文件并存（v0.1 不删），改版新开文件便于对照；
4. 上级审核走飞书云文档链接（互联网可阅读），不发文件。

### [模糊需求用选项式提问收束，不靠猜]

**核心**：需求方说「介绍如下xxx的一系列skill」这类带未展开占位、或「减负」这类程度词时，把歧义拆成 2-3 个单选题（每题给推荐项），一轮问清再动笔。

**来源**：本次减负需求含两处歧义（skill 范围指哪些；重头戏内容保留到什么程度），2 个选择题收束后 v0.2 一次过审。

**反例/边界**：歧义只有一处且有明显默认值时直接按默认做并注明即可，不必仪式化提问。

### [学员版减负三原则]

**核心**：面向学员的实操教程——①只讲「每个 skill 是谁、管什么」，不讲底层原理；②准备工作后置成结尾卡片，正文默认素材已备齐；③可选项（如 BGM）明确标「选配」，不混进必备清单。

**来源**：作者审核 v0.1 后提出的三条减负要求，本质是把「制作者视角」的教程改写为「使用者视角」；v0.2 落地后篇幅减约四成，一次过审。

**验证状态**：⚠️ 首次；下一篇同类教程正文完成后可升级为二次验证。

## 三、一句话结论

> 学员教程 = 完结项目的复盘反推 + 大纲层完成所有决策：事实从 git 里来，取舍在大纲上做，正文只是执行。
