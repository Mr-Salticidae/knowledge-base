# 交付前实测证伪律

> 首次记录：2026-07-14
> 来源：vpn-guard 全链路（见 [[2026-07-14_vpn-guard从工具到宣传片_全链路复盘_v1]]）中反复触发。
> 状态：**规律已确立**，凡"某方案应该能行"的假设都适用；跨平台/跨环境假设尤其。

---

## 一句话

**别交付未验证的"应该能行"。任何"X 应该有效"的假设，交付前先写最小探针去证伪它——尤其跨平台、跨环境、跨版本的行为假设，它们最爱静默失效。**

宁可花两分钟证伪，也不要交付一个"看起来对、实际不工作"的假方案——后者是最贵的债，因为它会带着你的信任一起交付出去。

---

## 本次三次触发（同一律，三种结局）

| 假设 | 实测 | 结局 |
|---|---|---|
| `TZ` 环境变量能让 Chrome 改时区（Linux/macOS 惯例） | 写探针 headless dump-dom：设 `TZ=Asia/Tokyo` 后 Chrome 仍报 `Asia/Shanghai` | **证伪** → 弃掉假方案，改 `tzutil` 切系统时区（真有效，端到端验证过） |
| Windows SAPI 能本地出中文 TTS | `System.Speech` 直接抛 `PlatformNotSupportedException`（沙箱） | **证伪** → 转 edge-tts（免费神经 TTS，测一句确认可用再全量） |
| git push 30MB 已完成 | `git ls-remote` 查远程分支为空 | **未完成** → 后台重推 + 核实远程文件树才收口 |

三次的共同点：**没有一次是"我觉得应该行就交付"**。头一次若不测，就会交付一个静默失效的时区启动器；那是最典型的、会带着信任一起交付的假方案。

---

## 口诀 / 判据

- **"应该"是信号词**：当你想说"这个应该能行/应该已经好了"，那正是该写探针的时刻。
- **最小探针**：能证伪就行，不必完整——一句 `dump-dom`、一次 `ls-remote`、一句 TTS。
- **跨环境必测**：平台惯例（POSIX 的 TZ）、版本行为、沙箱限制、缓存（如 `[TimeZoneInfo]::Local` 同进程缓存）——都爱在你没测的地方失效。
- **新起进程验证副作用**：改了系统级状态（时区/环境）要用**新进程**验证，别被当前进程的缓存骗了。

---

## 与姊妹律的分工（三段式）

同一"不轻信"的纪律，按时间点分三段，各有对口文档：

- **交付前**：本律——证伪假设（"这方案真的有效吗？"）
- **交付后**：[[Claude完成报告核查心法]] / [[Windows下编码与DPI的所见非真相]]——核查产出（"我做出来的真是对的吗？"，程序化字节级核查，不信肉眼）
- **事后**：[[复盘事实先行原则]]——冻结事实再判断（"我以为的成功真是成功吗？"）

三段合起来才是完整的"不把未验证当已验证"。
