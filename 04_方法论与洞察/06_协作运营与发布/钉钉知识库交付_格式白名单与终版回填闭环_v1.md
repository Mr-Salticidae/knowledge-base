  - **🔴 占位标注版**：每图替换为 `🔴【图N · 说明 ｜ 源文件：路径】`——对应「**复制粘贴进母文档**」通道（本档 2026-07-20 已记：粘贴场景公网直链同样失效）

- 定位修正：占位版不再只是"无法用直链时才退回的兜底"，而是**与直链版并列的通道适配版本**，两版一起交付、一起入库
- 附带约束：raw 直链依赖仓库**保持公开**且图片留在 main，仓库转私有会让已交付文档的图集体失效——这类文档要么锁定占位版，要么在转私有时同步替换

## 一、方法论沉淀

### [富文本平台交付：格式白名单先行，不等审核删改]

**核心**：往钉钉这类富文本知识库交付 Markdown，语法要按平台白名单**主动收敛**，而不是全量语法交上去等审核人手动删。本次白名单：修饰只用**加粗 + 代码块**；结构层面正文段落和有序列表可用；禁引用块、禁表格、禁多级小标题（并入正文加粗）。表格的信息改走图卡片或正文列点。

**来源**：终审 diff 的两处关键删改（表格删除、引用块转正文）都是格式层修改，与内容无关——这些返工本可以在交付前收敛掉。

**操作规则**：
1. 首次给一个新平台交付，交付后拿审核版 diff 一遍，把格式层删改提炼成该平台白名单；
2. 白名单一旦确立，后续交付按白名单写母版，变体试探（直链版/纯文字版/占位版）收敛为单一母版；
3. 白名单存进 skill / 规范文档，不靠记忆。

### [终版回填闭环：平台审核版是唯一真身]

**核心**：文档在平台上被人工修改并过审后，**平台版成为 source of truth**，必须立即下载回填 git 仓库（verbatim，不做任何「顺手优化」）；否则仓库版从此漂移，下次迭代会基于过时底稿返工。

**来源**：本次终审在钉钉编辑器里完成（开场白重写、加粗调整均发生在平台侧），仓库里 4 个变体没有一个等于过审版；不回填则真身只存在于钉钉。

**操作规则**：
1. 平台过审 → 当天下载 md → 与仓库最新变体 diff → verbatim 回填为「钉钉终版」文件；
2. diff 出的格式类修改沉淀进白名单，内容类修改沉淀进复盘；
3. 回填时发现审核版自身违反已定规则（如本次残留引用块），**不擅自改过审稿**，标记待办向审核人确认；
4. **内容类修改要反向同步回仓库底稿**（正文定稿/大纲展开稿），不能只留在「钉钉终版」文件里——否则下次基于底稿改版又会长出同样的问题。2026-07-20 实例：草稿里用内部编号引用其他章节（「实战 2」），终审改成实际章节标题（「自动化剪辑案例」），因为草稿编号和母文档最终编号不保证一致；这条修正回填进了正文定稿而不只是终版文件。

### [导入通道不明时用变体分治，通道定了就收敛]

**核心**：不确定平台导入行为（会不会自动抓图、认不认 Markdown）时，一文多变体各赌一条通道是合理试探；但通道和白名单确认后，变体只保留历史价值，母版收敛为一个，避免多副本漂移。

**来源**：单日 4 个变体提交是试探期产物；终版落地后它们全部被「钉钉终版」取代。

### [配图策略跟着交付通道走，通道变了先实测再选版]

**核心**（2026-07-22 作者澄清后更正）：钉钉有三条通道，能不能带图各不相同——**导入 md 文件：公网直链自动抓图，可带**；**钉钉文档 → 另一个钉钉文档复制粘贴：丢图**；本地文本粘贴：丢图。所以正解不是"退回占位版"，而是**把链路设计成 md 直接导入到目标位置，不要经过中转文档**；占位标注版只在无公网直链可用（私有仓库 / 图未入库）时才启用。

**来源**：2026-07-17 导入通道实测直链可行；2026-07-20 一度按"复制粘贴丢图"把占位版定为默认；2026-07-22 作者澄清真正丢图的是"钉钉→钉钉复制"这一步，导入本身没问题，遂改回直链并实测 14 张图零手工补图。归因错位的教训另立 [[变通方案不等于故障点_v1]]。

**操作规则**：
1. 每次交付先确认通道，并按"是否经过中转文档"判断能否带图；
2. 首选：md 带 GitHub raw 公网直链，**直接导入目标位置**；链接中文路径段百分号编码，交付前逐条 `curl -I` 确认 200（图须已 push 到 main、仓库公开）；
3. 无直链可用时才退回占位标注版，占位写 `🔴【此处插入图N xxx ｜ 源文件：docs/images/xxx/图N_xxx.png】`；
4. 视频与 zip 附件任何通道都带不进去，留 🔴 占位、导入后手动上传；
5. 通道结论当天写回 skill `dingtalk-doc-delivery`，不靠记忆。

### [教程文档的四段式体例：先给结果，再给工具，最后给能带走的资产]

**核心**：公司现行教程通行结构是**一、成品展示 → 二、所用工具 → 三、操作方法 → 四、总结与小技巧 → 五、附件**。三个易漏点：① 成品放在第一屏（读者先看到结果才有动力读下去）；② 「所用工具」要把**生成图片和视频的幕后工具**也列全（读者不用装的也列，注明"由 X 自动调用"），否则读者不知道成品里每样东西是谁做的；③ 文末挂一个 zip，把 skill / 模板 / 脚本这些**能带走的资产**打包，并写清包里没有、需要自己注册的在线账号。

**来源**：2026-07-22 作者指定按《鲸海拾贝-导演调度蓝图生视频教程》体例重做实战6；「补幕后工具」与「加附件压缩包」两条都是作者点名追加的，说明它们是体例里读者最在意的部分。

**操作规则**：
1. 幕后工具从**仓库真实脚本**核对（本次 faster-whisper / Pillow / FFmpeg 三样均取自 `transcribe.py`、`render_codex_tutorial.py`），不凭印象写；
2. 小技巧一节从项目复盘的"没有一次做对的"提炼，一条一句、加粗开头；
3. 附件 zip 放 `docs/attachments/` 随仓库提交；多处存在同名资产时先逐文件哈希比对再打包，避免打进旧版本；
4. 标题层级按文档定位：作为母文档子节并入时并入正文加粗，独立成页教程用「一、二、三」分节。

## 二、一句话结论

> 富文本平台交付 = 格式白名单 + 通道适配 + 终版回填：按白名单收敛格式、按通道决定图片走直链还是占位、按体例排成品-工具-步骤-总结-附件；过审后平台版是唯一真身，当天下载回填，仓库才不漂移。

## 如何使用

- 下次往钉钉母文档交付：直接按 skill `07_skill存档/dingtalk-doc-delivery` 的白名单写母版，跳过变体试探；
- 遇到新的富文本平台（语雀/Notion/飞书知识库等）：套用「交付→diff 审核版→提炼白名单→回填」同一套闭环。
