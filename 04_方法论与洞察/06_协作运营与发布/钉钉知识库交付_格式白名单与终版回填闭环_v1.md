## 一、方法论沉淀

### [富文本平台交付：格式白名单先行，不等审核删改]

**核心**：往钉钉这类富文本知识库交付 Markdown，语法要按平台白名单**主动收敛**，而不是全量语法交上去等审核人手动删。本次白名单：修饰只用**加粗 + 代码块**；结构层面正文段落和有序列表可用；禁引用块、禁表格、禁多级小标题（并入正文加粗）。表格的信息改走图卡片或正文列点。

**来源**：终审 diff 的两处关键删改（表格删除、引用块转正文）都是格式层修改，与内容无关——这些返工本可以在交付前收敛掉。

**操作规则**：
1. 首次给一个新平台交付，交付后拿审核版 diff 一遍，把格式层删改提炼成该平台白名单；
2. 白名单一旦确立，后续交付按白名单写母版，变体试探（直链版/纯文字版/占位版）收敛为单一母版；
3. 白名单存进 skill / 规范文档，不靠记忆。

### [终版回填闭环：平台审核版是唯一真身]

**核心**：文档在平台上被人工修改并过审后，**平台版成为 source of truth**，必须立即下载回填 git 仓库（verbatim，不做任何「顺手优化」）；否则仓库版从此漂移，下次迭代会基于过时底稿返工。

**来源**：本次终审在钉钉编辑器里完成（开场白重写、加粗调整均发生在平台侧），仓库里 4 个变体没有一个等于过审版；不回填则真身只存在于钉钉。

**操作规则**：
1. 平台过审 → 当天下载 md → 与仓库最新变体 diff → verbatim 回填为「钉钉终版」文件；
2. diff 出的格式类修改沉淀进白名单，内容类修改沉淀进复盘；
3. 回填时发现审核版自身违反已定规则（如本次残留引用块），**不擅自改过审稿**，标记待办向审核人确认；
4. **内容类修改要反向同步回仓库底稿**（正文定稿/大纲展开稿），不能只留在「钉钉终版」文件里——否则下次基于底稿改版又会长出同样的问题。2026-07-20 实例：草稿里用内部编号引用其他章节（「实战 2」），终审改成实际章节标题（「自动化剪辑案例」），因为草稿编号和母文档最终编号不保证一致；这条修正回填进了正文定稿而不只是终版文件。

### [导入通道不明时用变体分治，通道定了就收敛]

**核心**：不确定平台导入行为（会不会自动抓图、认不认 Markdown）时，一文多变体各赌一条通道是合理试探；但通道和白名单确认后，变体只保留历史价值，母版收敛为一个，避免多副本漂移。

**来源**：单日 4 个变体提交是试探期产物；终版落地后它们全部被「钉钉终版」取代。

### [配图策略跟着交付通道走，通道变了先实测再选版]

**核心**：「公网直链自动抓图」只在**导入 md 文件**通道成立；**复制粘贴**通道图片视频全部丢失，占位标注版（占位内带仓库源文件路径，补素材按图索骥）才是默认母版。母版收敛是**相对交付通道**而言的——通道一变，当初被收敛掉的变体可能回归为唯一正解，先花一次实测确认再批量转换。

**来源**：2026-07-17 导入通道实测直链可行；2026-07-20 改走复制粘贴，图/视频粘不进钉钉，占位版回归为默认。

**操作规则**：
1. 每次交付先确认本次通道：导入文件还是复制粘贴；
2. 复制粘贴通道 → 占位标注版，占位写 `🔴【此处插入图N xxx ｜ 源文件：docs/images/xxx/图N_xxx.png】`；
3. 通道结论当天写回 skill `dingtalk-doc-delivery`，不靠记忆。

## 二、一句话结论

> 富文本平台交付 = 格式白名单先行 + 终版回填闭环：交付前按白名单收敛格式，过审后平台版是唯一真身，当天下载回填，仓库才不漂移。

## 如何使用

- 下次往钉钉母文档交付：直接按 skill `07_skill存档/dingtalk-doc-delivery` 的白名单写母版，跳过变体试探；
- 遇到新的富文本平台（语雀/Notion/飞书知识库等）：套用「交付→diff 审核版→提炼白名单→回填」同一套闭环。
