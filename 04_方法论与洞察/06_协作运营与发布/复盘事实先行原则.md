# 复盘事实先行原则

> 入档:2026-05-15
> 触发:`灵光一现+风花雪月双题复盘` 整合时,把"我喜欢的方案"误写成"获奖方案",次日重读才发现并校准
> 主线作品来源:`03_prompt模板库/02_案例复盘/灵光一现+风花雪月双题复盘.md`(§元复盘段)
> 性质:写作姿态原则,跨复盘 / 数据分析 / AI 协作场景通用

---

## 一句话

**写复盘文档时,先把事实(结果、票数、排名、数据)冻结在文档顶部的元数据里,再开始写赢点和对比**。否则会把"我喜欢的设计"潜意识等同于"成功的设计"——这是一种自我归因偏差。

---

## 案例:一次踩坑

2026-05-15 整合「灵光一现+风花雪月」双题复盘时:

- 跳蛛先生写的源文档标题:`MPB 灵光一现+风花雪月复盘`(无"获奖")
- 内容里两个 H1 写成:`「灵光一现」获奖图复盘` · `「风花雪月」获奖图复盘`
- 主对话 Claude 整合时:**直接采纳"获奖"措辞**,新建 `灵光一现+风花雪月获奖图复盘.md`,把跳蛛先生的方案全程当"获奖图"分析,写出 11 条方法论

**事实是**:跳蛛先生的两个方案**票数靠前但未达获奖线**,获奖的是同期对手作品(灵光一现的牛顿苹果灯泡头、风花雪月的雪夜樱花月 + 粉色山林老夫妇)。

**结果**:文档发布后,跳蛛先生重读时才察觉错误。需要做一次全仓库的校准:
- 文件重命名(获奖图复盘 → 双题复盘)
- 14 处 wiki 引用同步更新
- 主文档大改(把"我的方案"和"获奖图"分开表述)
- 跨双仓库 commit + push

**这一切本来不必发生** —— 只要在整合前先问一句"你的方案获奖了吗?"。

---

## 为什么会发生这种偏差

### 偏差源头

1. **叙事惯性** —— "复盘"这种文档体裁默认指向"成功案例",大脑自动补完"赢点叙事框架"
2. **概念向偏差** —— 我的方案确实有概念巧思(11 条方法论是真实的),容易让人误以为"巧思 = 获胜"
3. **缺事实锚** —— 源文档没在顶部写"获奖结果",叙事就接管了事实
4. **AI 整合放大** —— AI(包括 Claude)倾向于沿用源文档措辞,如果源里有歧义会被放大成断言

### 偏差的实际成本

- **创作判断被污染** —— 把"票数靠前"的设计当"获奖"叙事,会错过真正的"路径选择"复盘
- **跨文档传染** —— 一篇文档错了,所有引用它的方法论都跟着错(本次涉及 14 处引用)
- **长期资产污染** —— 知识库的方法论会被反复引用,事实错误会被反复引用
- **元复盘失效** —— 没看清"为什么我的方案没获奖"就开始庆祝赢点,等于丢掉了最值钱的复盘点

## 如何使用

### 写新复盘文档时:四要素 frontmatter

每篇复盘文档**顶部的引用块**必须明确写出**事实四要素**,再开始任何分析:

```markdown
> 入档:YYYY-MM-DD
> **事实结果**:
> - 比赛/榜单/活动:[活动名]
> - 我的作品:[作品名] / 排名:第 X / 票数:N / **获奖与否:✅ 获奖 / ❌ 未获奖 / 🟡 入围未获奖**
> - 获奖作品:[作品名 + 作者](如非本人)
> 性质:...
```

只有在这个框写完之后,才可以进入"我的方案赢点"或"获奖图分析"。

### 写"赢点"前的三问

每个"赢点"动笔前,在心里(或在文档草稿里)回答:

1. **这个赢点导致获奖了吗?** —— 是 → 赢点;否 → 改用"设计亮点"或"我的方案要点"
2. **如果换一个评委,这个赢点还成立吗?** —— 是 → 客观赢点;否 → 主观偏好,要标注
3. **同期对手的获奖图有这个赢点吗?** —— 没有 → 你和对手走了不同路径;有 → 你和对手共有,只是执行差异

### 措辞校准对照表

| 容易写错的 | 校准为 |
|---|---|
| "获奖图复盘" | "双题复盘" / "案例复盘" / "本期方案复盘" |
| "我们的获奖方案" | "我的方案" / "我喜欢的方案" |
| "图本身的赢点" | "我的方案设计要点" / "我的方案设计巧思" |
| "获奖 prompt 完整存档" | "我的方案 prompt 完整存档" |
| "同期高分对手" | "本期获奖图" / "实际获奖作品" |
| "双题双中" / "三张同时获奖" | "双题方案 + 同期获奖图对比" |
| "我们拿了奖" | "我的方案入围 / 未获奖" |

## 适用范围扩展

本原则不只适用于"获奖复盘",所有"事实-叙事"会被混淆的场景都适用:

### 场景 1:数据复盘

- ❌ "用户喜欢这个功能" → ✅ "这个功能的留存率是 X%,日活贡献度 Y%"
- 把"我喜欢的解读"和"数据真正说的"分开

### 场景 2:用户访谈整理

- ❌ "用户觉得 X 很重要" → ✅ "10 个用户里 3 个提到 X,其中 2 个是被引导提及"
- 把"我想听到的话"和"用户真的说了的话"分开

### 场景 3:AI 协作产出复盘

- ❌ "Cowork 完美交付了任务" → ✅ "Cowork 交付 X,其中 A 部分需返工,B 部分自我评分"
- 把"协作整体顺利"和"具体每步的实际表现"分开

### 场景 4:项目复盘

- ❌ "项目成功" → ✅ "项目按时上线 / 用户 N 万 / 目标完成度 X%"
- 把"达成感"和"客观结果"分开

### 场景 5:本人作品成绩复盘

- ❌ "这条作品爆了" → ✅ "这条作品播放 N / 涨粉 X / 互动率 Y%"
- 把"主观爽感"和"客观数据"分开

## 元层认知:为什么"未获奖也值得复盘"

这次踩坑的元 insight 之一:**未获奖的方案常常比获奖方案更值得复盘**。

- **获奖方案**:只能看到"什么有效",看不到"路径选择对错"
- **未获奖方案 + 获奖方案对比**:能看到"我选了概念路径,对手选了视觉奇观路径,本期评价场景下后者赢"——这才是路径级的复盘

把"未获奖"当作"失败"会让人本能回避复盘。但只有在**承认未获奖的事实**之后,才能问"我选错了路径吗?",才能产出真正可复用的"路径选择决策树"。

参考 [[红海主题的三条破局路径]] —— 这条方法论就是从"我的方案没获奖"的对比中提炼出来的,如果没承认事实就只会写"获奖图复盘",这条决策树永远写不出。

---

## 反例:何时不必这么严格

- **私人笔记 / 草稿** —— 自己看的,叙事先行没事,后期整理时再做事实校准
- **创作日志 / 灵感记录** —— 这类是"主观体验记录",事实锚反而冗余
- **未完成的项目记录** —— 还在跑的项目,事实未到位,可以先用占位符
- **明确标注为"我的偏好"的内容** —— 比如个人 IP 视觉系统的色板偏好,本来就是主观

**核心判断**:**这份文档将来会被(自己或别人)当作客观事实引用吗?** 是 → 用本原则;否 → 不必。

---

## 一条机制层补丁:Cowork 整合复盘文档时的护栏

**针对 AI 协作场景**:Cowork / 副会话整合复盘文档时,**必须先用 AskUserQuestion 确认事实**,再开始整合,具体三个问题:

1. **结果**:这个案例的客观结果是什么?(获奖/未获奖、播放/涨粉、目标达成/未达成)
2. **对照组**:这个案例的对照组是什么?(对手作品、同类型其他案例、上一版本数据)
3. **你想沉淀的角度**:你希望从"赢点"还是"路径选择"还是"踩坑"哪个角度复盘?

**不许跳过这三问直接整合**——这次踩坑就是因为跳过了第 1 问。
