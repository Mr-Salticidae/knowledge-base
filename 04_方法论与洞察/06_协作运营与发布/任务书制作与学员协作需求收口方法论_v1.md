# 任务书制作：把合作需求变成可执行的学员任务

> 入档：2026-07-20
> 来源：对“赛博废土武侠 · 复刻案例素材及教程制作”任务书的归档与抽象（工作区文件：`{AIGC工作站}/tutorial/赛博废土武侠 · 复刻案例素材及教程制作 · 任务书.md`）
> 验证状态：✅ 首个任务书全流程落地（2026-07-20 发书「当代年轻人精神状态检测」→ 当日交付 → 07-21 验收/裁决/结算收口）；落地暴露三处条款缺口，已回写模板（见「六、落地回验」）

---

## 事实记录（不可修改区）

- 任务：把一次“教程合作 + 案例复刻 + 学员外包”的需求，整理成可直接转发给学员的任务书。
- 典型对象：案例实战教程、短视频复刻、素材补齐、教程补写等外包式协作。
- 目标：让接收方一眼知道“做什么、参考什么、何时交、多少钱、怎么交”。
- 现状：以“赛博废土武侠复刻案例素材及教程制作”作为首个可复用样板，后续同类项目可直接套用。

## 一、方法论结论

### [任务书不是需求说明，而是“可执行协议”]

**核心**：一份好任务书，不是把想法写满，而是把模糊需求压成“接收方能立即执行”的明确信息集合。它要回答四件事：

1. 目标是什么；
2. 参考什么；
3. 什么时候完成；
4. 交付什么、怎么结算。

**适用场景**：教程合作、案例复刻、素材外包、学员助力型内容制作。

### [任务书的最小结构，建议固定为七段]

1. 一句话总结：概括目标、内容和预算；
2. 具体要求：拆成编号条目，尽量可执行；
3. 对标视频 / 对标素材：明确参考对象；
4. 时间要求：写清完成节点；
5. 作者公开的工作流：给出可参考的工具、资料、流程；
6. 交付流程：说明最终交付形式与渠道；
7. 付款说明：写清金额、时间、税费和到账情况。

### [先写“交付结果”，再写“执行细节”]

**核心**：任务书必须先聚焦产物，而不是先写背景。否则接收方会把注意力放在“你想要什么风格”这种模糊描述上，而不是明确可执行的事项。

**操作规则**：
1. 先写“要做什么产物”；
2. 再写“要达到什么规格”；
3. 最后写“参考什么、什么时候给、怎么结算”。

### [模板化是为了减少沟通成本，不是为了变成空壳文案]

**核心**：任务书模板应该足够结构化，但仍保留空位，让每次项目能替换主题、时间、参考素材和预算。模板是“骨架”，不是“硬套”。

**反例/边界**：如果一个项目连参考素材都没有、时间节点都没定，先不要急着写成正式任务书；先把这些信息补齐，否则任务书只会变成一份空泛的说明。

## 二、可复用模板

```markdown
# [主题] · [项目名称] · 任务书

## 零、一句话总结

制作一期[内容类型]，目标是[目标]，费用[金额]元。

## 一、具体要求

01
按照[对标对象]复刻制作[时长/规格]的[产物类型]

02
参考[参考资料/文档]的格式与文风，同步制作相关[教程/说明]

03
素材将用于公域宣传：不得含个人特征（真实姓名/昵称/联系方式入画入文）与第三方品牌标识；产品内需要人设名时，使用无指向性的中性称呼

04
[补充要求]

## 一之二、范围变更

若素材实况与本任务书假设不符（如已有成品、素材缺失、规格不符），请先回报发布方，经确认后调整范围；单方面改述任务范围不自动生效。

## 二、对标视频

[请至[平台/文档]查看附件《[文件名/链接]》]

## 三、时间要求

[时间节点]之前完成

## 四、作者公开的工作流

[相关工具/平台/链接]

## 五、以往对标参考文档

[参考链接 1]

[参考链接 2]

## 六、交付流程

[交付说明]

## 七、付款说明

**薪酬：**[金额]元

**结算周期：**验收通过后纳入每月[X]号左右集中打款

**税：**[比例]

**实际到手：**[金额]元
```

> 模板使用铁律：发书前每个字段要么填实、要么删除，**不留空**——留空字段等于让承接方自行想象（「转账时间」留空曾导致承接方默认预期即时到账）。

## 三、写任务书时的建议顺序

```text
① 先确认项目主题与目标产物
② 再确认参考素材与关键要求
③ 再写时间节点与交付形式
④ 最后补齐费用、付款说明和备注
```

## 四、交付载体三级(2026-07-20 补)

同一份任务书,按发布场景选载体,当天实测走完全部三级:

1. **文件转发**(默认):md/截图直接发微信、钉钉。最快,但引用的本地素材对方拿不到;
2. **链接嵌入版**(一次性转发):参考文档全文 + 配图整体嵌进一个网页链接(Claude artifact 或同类),对方点开即看全,无需索取附件;
3. **站内常设版块**(正式渠道):`tiaozhuxiansheng.com/tasks/` 任务书版块,2026-07-20 正式起刊。发布/收官/撤稿 = 增删改一个数据文件(承载工程见 [[站内新版块五件套模式_三次复用与起刊两坑_v1]])。

**选择判据**:偶发单份合作 → 链接嵌入版;持续对外招募 → 站内版块。

**两条实操律**(嵌入版/上站前必查):

- **本地路径不出门**:任务书正文里 `E:\...` 这类本地引用,对接收方等于没写;对外前必须换成对方可达的形式——在线链接,或直接把内容嵌进交付物。
- **嵌入版工艺**:配图先压缩再内嵌(实测 14 张 8.8MB PNG → 1400px JPEG q82 ≈ 1.3MB,整页 ≈ 1.7MB,打开不卡);文档类页面**固定单主题**(纸面浅色),否则查看端深浅色模式打架,浅色字被洗在白底上不可读。

## 五、落地回验（2026-07-21 · 首个任务书走完发书→交付→验收→裁决→结算全流程）

**案情**：「当代年轻人精神状态检测」教程素材补充任务书（150 元，截止 2026-07-20 17:00）按七段结构发出，学员当日交付（教程稿 + 9 图 + 30s 视频 + 可运行网页成品）；学员不用 GitHub，由 Claude 代理对接（收件 → 技术验收 → 落库 → 格式白名单收敛）。全流程走通，但暴露三处条款缺口，均已回写模板：

### [缺口一 · 范围条款基于假设而非素材实况，给了承接方单方改述的空间]

**事实**：任务书写「根据提供的附件文档和配图，补充教程内容素材」，但学员实际持有**完整可运行成品**——范围假设与素材实况错位。学员随附「任务书调整版」单方面把范围改述为「核验成品 + 整理教程」，需要作者事后裁决（本次作者认可，按调整后范围验收）。

**规则**：① 发书前先开包核验素材现状（有无成品、完成度几何），范围条款写实况不写假设；② 模板补「范围变更」条款——素材实况与任务假设不符时**先回报、经发布方确认后调整**，承接方单方改述不自动生效。本次裁决流程本身是对的（代理方标注「待裁决不计入完成」→ 作者确认制），沉淀为标准动作。

### [缺口二 · 公域红线没有前置进任务书，交付后才发现返工]

**事实**：交付素材含学员个人特征人设「陈医生」（姓氏），任务书无中性化条款，作者审核时才发现，返工纯净版 + 重拍 2 图 + 正文改写。本库已两案同构：镜像自我（合作方/展会标识，交付后去除）→ 本次（个人人设，交付后去除）。**可预见的返工点就该是标准条款。**

**规则**：模板「具体要求」段加固定红线条——素材将用于公域宣传，不得含个人特征（真实姓名/昵称/联系方式入画入文）与第三方品牌标识；产品内需要人设名时用无指向性的中性称呼。

### [缺口三 · 付款说明有方式无周期，承接方默认预期即时到账]

**事实**：任务书付款段写了金额、方式（微信直转）、税率，但「转账时间」空缺；实际结算是**每月 15 号左右集中打款**，验收后需要额外补充说明。

**规则**：结算写周期不写单日——模板「转账时间」字段改为「结算周期：验收通过后纳入每月 X 号左右集中打款」。模板字段不留空：发书前每个字段要么填实、要么删除，留空字段=给承接方留下自行想象的空间。

**顺带回验（交接口头规则的效力）**：发书时口头补充过三点交接说明（对标提炼骨架不抄句式 / 格式产物由仓库统一收敛 / 配图真实性标注）。回验结果：内部术语零残留（有效）；但学员引言与对标文档开头句式仍然同构（「如果你……离……还有多远？」），作者过审、并入母文档前连读再核——说明**行文层规则口头转述效力有限**，重要的行文红线应写进任务书或附样例对照，参见 [[范文参考骨架律_v1]]。

## 五之三、测评类批量发布回验（2026-07-24 · 一次发三份工具测评任务）

**案情**：一次性发布三份同构的「工具评测类出教程」任务（OJO / Flova / 问小白 5 Pro）。三份共享同一交付规范，只有测评对象、价格、账号情况不同——批量场景把两条此前没显式化的条款逼了出来，均补进模板。

### [新条款一 · 按测评对象「模态」定价，不是一口价]

**事实**：图像 / 视频类工具测评定 150 元，LLM（问小白 5 Pro）定 100 元。差价不是随意的——LLM 成品以文本 / 对话为主，出稿工作量与配图量低于图像 / 视频类。

**规则**：测评类任务的报价**跟测评对象的模态走**（文本 < 图文 < 视频），同批发布时把差价依据写进任务书（如「本任务测评对象为 LLM，成品以文本为主，故 100 元；另两份图像 / 视频类为 150 元」），避免承接方觉得厚此薄彼。

### [新条款二 · 测评付费工具，必须写明「账号谁出、要不要自付」]

**事实**：测评对象常是**付费**工具 / 会员。若任务书不写账号来源，承接方会默认要自掏腰包买会员——这会劝退接单，或事后扯皮。本批统一加「账号与开销」段：**制作所需会员 / 账号由发布方默认提供，测试与出稿零自付开销**。

**规则**：凡测评 / 试用**付费产品**的任务，模板固定加一段写清 ①账号 / 会员谁提供 ②承接方要不要自付。「零自付」是重要的招募卖点，别让它藏着。

### [复用确认 · 规范与任务书分离，任务书只写「任务特定信息」]

三份任务书正文的格式要求（成品展示 / 步骤流程 / 总结三节、标题命名、上传知识库）**一律指向站内规范页 `/tasks/spec/`，不再逐份重写**；任务书本体只留任务特定信息：测评对象、价格、DDL、账号。这是 [[钉钉知识库交付_格式白名单与终版回填闭环_v1]]「格式规范单独立、内容只填差异」在任务侧的同构落地。

> 承载细节:任务书正文里指向规范页的站内链接走**相对路径** `[规范](../spec/)`,而非写死 `/tasks/spec/`——md 正文注入不了 base 变量,相对路径才能在双部署目标下都解析对（见 [[双部署目标的base路径陷阱_根路径拼出双斜杠_v1]]「markdown 正文里的站内互链走相对路径」）。

## 六、一句话结论

> 任务书的作用，不是“把需求写得漂亮”，而是把需求压成一份“能被执行、能被审核、能被结算”的可交付协议。
