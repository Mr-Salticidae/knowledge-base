# 角色卡 vs 主办方意图 · 协作元方法论 · v1

> 入档：2026-05-20
> 触发：副会话 #4 涂图儿+良伞会话——Claude 反复在"自推角色卡" vs "主办方实际审美" 之间摇摆，导致涂图儿全程撞墙
> 性质：**跨项目协作元方法论**——任何"外部审稿方 + Claude 协助" 的项目都适用
> 关联：[[../../复盘记录/复盘笔记_涂图儿良伞_悠船niji5_Nano极简_2026-05-20]] / [[Claude完成报告核查心法]] / [[识别工具天花板的时机]] / [[Cowork协作的接口文件模式]]

---

## 一句话

**Claude 推导的"角色卡设定" 是工作锚点，不是合规标准。当工具反复输出"意外解" 且主办方未明确否决时，先送审让主办方拍板，不要在 Claude 脑补的"合规标准" 里反复消耗用户时间和资源。**

---

## 一、触发场景

### 1.1 复盘案例（涂图儿撞墙史）

涂图儿（PB 冠军 #4 · 白虎兽人女）形象图制作中，Claude 反复在两套标准之间摇摆：

- **Claude 自推的"角色卡设定"**：黑色 ArtDeco 礼服 + 翡翠绿 + 真兽人
- **主办方实际审美锚点**（事后才看清）：白色为主 + 银白头饰 + 接受双形态意外解

工具反复输出"意外解"（银发美人 + 伴生白虎双形态）—— Claude 把这视为"撞墙"，硬掰回角色卡方向，**全程 5-6 轮 niji 5 重做都死磕"真兽人 + 黑礼服"角色卡** → 全部撞墙。

niji 5 v5 释放约束后（只锚定"虎元素 + 黑色 + 复古高贵"）→ 立刻出"银发美人 + 伴生白虎" 双形态——和最初被驳回的图本质上是同方向。送审 → 主办方搁置（对涂图儿整体兴趣已减）。

**根因诊断**：撞墙不是工具或 prompt 的问题，是 **Claude 把角色卡当成了合规标准、不愿意接受工具反复输出的"意外解"**。

### 1.2 对照案例（良伞胜利路径）

良伞（PB 冠军 #5 · 人形山海经文鳐鱼男）一次出种子图通过——核心是**释放角色卡约束**：

- 角色卡列了"鸟翼 / 鳞片 / 怀抱文鳐鱼" 等细节
- 工具 niji 5 v4 跑出"袍上鱼纹替代怀抱" 的意外解
- Claude 没有硬掰回"必须怀抱" → 送审 → 主办方通过

**核心胜利**：**没有把角色卡当成合规标准**。

## 二、核心规律

### 2.1 角色卡的真实身份

| 维度 | 误解 | 正解 |
|---|---|---|
| 来源 | "主办方提供的合规标准" | **Claude + 上一个会话推导的** |
| 验证状态 | "已经被认可" | **主办方对此从未正式确认** |
| 使用方式 | "必须遵守的硬指标" | **工作锚点，未经验证的假设** |
| 冲突时 | 角色卡优先 | **主办方意图优先** |

**角色卡只是 Claude 的工作起点，不是合规标准**。

### 2.2 主办方反馈中的"硬指标 vs 软指标"

主办方驳回理由通常模糊（如"和设定差别太大"）。Claude 容易把模糊反馈解读为"必须严格回归原始角色卡"——**这是错的**。

正确解读路径：

```
模糊反馈 → 拆解"设定" 具体指什么 → 找主办方真正认可过的视觉锚点 →
保住已认可锚点 → 其他元素允许工具自由发挥
```

### 2.3 工具反复输出"意外解" 的意义

**当工具在多轮重做后反复输出某种与角色卡不同的形态**（如双形态、特定颜色组合、特定构图），这往往不是工具的"问题"，可能是：
- 主办方真正想要的形态（角色卡推导错了）
- 训练数据里的天然语义组合（工具的本能往往合理）
- 用户审美的真实倾向（数据驱动的暗示）

**正确做法**：**先送审让主办方拍板**，不要硬掰回角色卡方向。

---

## 三、操作规则

### 规则 1 · 角色卡是假设，不是合规标准

任何 Claude（或上一个会话）推导的角色卡设定，在主办方明确确认之前，**都是未经验证的工作假设**。

- ✅ 把角色卡作为"工作起点" 使用
- ❌ 不要把角色卡作为"合规标准" 强守
- ✅ 当工具反复反弹角色卡方向时，**先质疑角色卡，不要硬掰工具**

### 规则 2 · 模糊反馈要拆解，不要硬扩张

主办方说"和设定差别太大" 时：

**不要做**：把"设定" 默认理解为"原始角色卡的所有细节"
**要做**：
1. 拆解"设定" 具体指什么（核心物种？关键颜色？主体形态？）
2. 找出主办方真正认可过的视觉锚点（已通过的图、明确表态过的元素）
3. 优先保住这些已认可锚点
4. 其他元素允许工具自由发挥

### 规则 3 · 意外解 + 未明确否决 = 值得送审

**判断公式**：

```
工具反复输出某种形态 + 主办方未明确否决该形态 = 值得送审的方案
```

不要在 Claude 自己脑补的"合规标准" 里反复消耗用户时间和资源。

### 规则 4 · 接受"意外解" 不是放弃质量，是让真实信号说话

工具的"意外解" 不是它的"bug"，而是它根据训练数据 + prompt 的"最优概率输出"。当多轮 prompt 调整后输出仍指向同一方向，**这是个强信号**——可能是：
- 主办方审美的隐性倾向
- 题面词汇的天然语义
- 训练集里的合理组合

把这个信号送审，让主办方判断——比 Claude 自己脑补"应该是什么样" 更接近真实。

## 四、检查清单（Claude 自查）

当 Claude 在某个项目里反复撞墙时，先做以下自查：

- [ ] 我正在守的"标准" 是 Claude 自推的，还是主办方明确确认过的？
- [ ] 我把主办方的模糊反馈扩张解读了吗？
- [ ] 工具在多轮重做后是否反复输出同一种"意外解"？
- [ ] 这个"意外解" 是否被主办方明确否决过？
- [ ] 我是不是在"防御性写作" —— 想把所有可能出错的地方都堵死？

如果其中两条以上为是，**停下来，先送审一个"意外解" 版本**，让主办方拍板。

[[识别工具天花板的时机]] 讲：**当工具在某层反复触顶时，让 LLM 接管该层** —— 这是"工具撞天花板" 的应对。

两条同源：**识别"自己撞了天花板" 的能力是 AI 协作中最重要的元能力之一**。

### 与 [[Claude完成报告核查心法]] 的关系

[[Claude完成报告核查心法]] 讲：**Claude 的"已完成" 判定常基于代码路径，但用户看到的是运行时表现**。

本方法论是同源的另一个角度：**Claude 的"合规判定" 常基于自推角色卡，但主办方看到的是实际审美匹配**。

更上位的元规律：**Claude 的"自我判断" 有结构性局限——基于内部逻辑而非外部验证。任何"自我判断" 都需要被外部信号校准**。

### 与 [[Cowork协作的接口文件模式]] 的关系

接口文件模式讲："所有决策、理由、约束都通过显式 markdown 文件互传"。

本方法论补充：**Claude 推导出的"角色卡"（工作锚点）不应在文档里被表述为"合规标准"**。在共创者 brief 里要明确：
- 哪些是"主办方已确认的硬指标"
- 哪些是"Claude 推导的工作锚点"
- 两者冲突时主办方优先

未来给副会话的 brief 应该按这个分类标清楚。

## 六、防御性写作 vs 创造性合成 · 元教训

涂图儿撞墙的元层根因还有一条：**Claude 的"防御性写作" 把创造性合成压制了**。

精密的 prompt（堆叠遮挡关系/位置锚点/光影统一/反向禁止）本质是"想把所有可能出错的地方堵死"。但生成式 AI 工具的特点是"在弹性约束下的创造性判断"，防御性写作反而把创造性压制了。

**详细规律见 [[方法论笔记_AI形象图工作流分层_niji5+Nano]] §十二 · Nano 极简 prompt 原则**。

本方法论与那条**同源**：
- §十二 讲"对工具的弹性约束 vs 防御性精密 prompt"
- 本方法论讲"对主办方的灵活解读 vs Claude 自推合规标准"

两条规律的共同元层：**Claude 倾向于过度防御，反而压制了真实信号**。

---

## 七、适用场景

### 适用 ✅

- 任何"外部审稿方 + Claude 协助" 的项目
- 角色形象图项目（主办方/客户提供模糊要求）
- 文案创作项目（编辑/客户审稿）
- 视觉设计项目（用户/品牌方决策）
- 工程实现项目（产品经理/团队决策）

### 不适用 ❌

- 没有外部审稿方的纯个人创作（Claude 与跳蛛先生单独协作时，跳蛛先生本人就是审稿方，本方法论的"主办方" 直接换成"跳蛛先生" 即可）
- 已经有明确硬指标的合规项目（如法律文档格式 / API 接口规范 等——这种情况下硬指标确实是硬指标）

## Claude 推导的工作锚点（参考性，可调）
