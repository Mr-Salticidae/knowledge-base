# 公开前审计报告 · 2026-05-25

目标:评估本知识库是否适合切换为 GitHub public 仓库,并列出公开前需要处理的风险点。

## 总体结论

**建议公开,但先做一轮轻量清理。**

当前仓库的主体是 Markdown 方法论与少量 Python 示例代码,没有跟踪大型图片、视频、音频素材。扫描未发现真实 API key、GitHub token、AWS key 等高危密钥。主要风险不是"泄密",而是:

1. 仓库仍带有私有仓库时代的自动同步脚本和说明。
2. 文档里保留大量本机绝对路径,会暴露工作区结构,也会降低外部读者可复用性。
3. 暂无开源许可协议,公开后别人不知道能否转载、改写或商用。
4. 代码资产多为项目内脚本镜像,对外需要标注为"参考样例",避免被误解成开箱即用工具。

## P1 · 建议公开前处理

### 1. 增加 LICENSE

当前没有 `LICENSE` / `NOTICE` 文件。建议选择:

- `CC BY-NC 4.0`:适合"同好参考、署名转载、非商业使用"。
- `CC BY 4.0`:适合更开放的转载、改写和商业使用。
- 代码如需单独授权,可在 `06_代码/` 下补充 MIT 说明;否则统一按文档许可处理。

推荐:先用 **CC BY-NC 4.0**。它更符合"供同好朋友参考使用"的语境。

### 2. 私有同步脚本与说明

以下文件是私有仓库时代的维护脚本,公开后容易造成误解:

- `_auto_sync.ps1`
- `_register_schedule.ps1`
- `_setup_git.ps1`
- `GIT_SETUP_README.md`

处理建议:

- 如果公开仓库只做知识分享:移除这些脚本和说明。
- 如果仍想保留自动同步经验:移动到 `06_代码/仓库维护脚本示例/`,并改写成通用模板,去掉"私有仓库"叙述和本机路径。

### 3. 本机绝对路径

仓库中有大量 `{AIGC工作站}/...`、`{旧工作区}/...`、`{运行环境}/...` 路径。它们不是密钥,但会暴露工作区结构,也会让外部读者误以为必须有同样目录。

高频位置:

- `README.md`
- `03_prompt模板库/03_流程规范/交接文档书写规范.md`
- `04_方法论与洞察/*`
- `05_视觉系统/*`
- `06_代码/*`

处理建议:

- 方法论文档里保留必要的"来源路径"时,改成相对语义,如 `项目工作区/30_伊恩...`。
- README 的用法规范里,把"裸路径"规则改为"本地私库可用裸路径,公开版建议使用相对说明"。
- 代码里的硬编码素材路径改成示例变量或注释。

### 4. 仓库定位说明

当前 README 更像内部 Obsidian MOC。公开后建议在顶部增加一段给外部读者的说明:

- 这是什么:一个 AIGC 创作方法论知识库。
- 适合谁:AI 图像/视频/音乐创作者、Obsidian 笔记用户、AIGC 工作流研究者。
- 如何阅读:从 README 的 MOC 进入,不需要复刻本机目录。
- 许可与署名:引用时请保留作者与链接。
