# 每日定时任务自动更新内容板块:首次自动化生产落地

> 一句话:自动化内容生产的安全边界 = **只让机器碰数据文件,人先手动跑通一遍,并给它一个「不发」的出口**。

> **补丁(2026-07-05)**:调度方案已于 07-03 迁移为 GitHub Actions 云端 workflow,本档所记的本机定时任务从未经历无人值守运行;「首跑待观察」的三个点由云端方案接续验证,结果与连环踩坑见 [[云端定时内容生产连环坑复盘_全绿不等于已发_v1]]——其中对本档「人先跑通一遍再交给定时器」补充了重要边界:手动发布会让自动路径被幂等检查跳过,造成「已验证」假象。

## 事实记录(不可修改区)

- 项目:蛛网之上「AIGC 快讯 · AIGC Daily」板块(`/news/`,每日一期 AIGC 新闻精选)
- 需求提出 → push 上线:2026-07-02 上午,单次会话内完成(commit `dcf27c4`,7 文件 +482 行)
- 首期内容:第 1 期 7 条真实新闻(头条 Gemini Omni Flash / Claude Sonnet 5),全部带来源 URL
- 自动化:本机 Claude 定时任务 `aigc-daily-news`,cron `30 9 * * *`(平台固定 jitter +256s,实际 ~9:34 触发)
- **验证状态:手动全流程已跑通并上线;定时任务已挂载但截至归档时(2026-07-02)尚未经历过一次真实无人值守运行**——首次自动运行在 2026-07-03 早
- 作者体验评价:「非常完美」(针对板块呈现与首期内容;非针对无人值守运行)
- 数据来源:会话执行记录 + git log + 定时任务清单

## 做了什么

**架构:数据与呈现彻底分离。**

- 数据:`src/data/news/YYYY-MM-DD.json` 一天一期(headline + items[],每条 title/summary/category/source/url/featured),schema 注释写在 `src/lib/news.mjs` 头部,让未来的执行者(人或 AI)有单一参照
- 呈现:Astro 构建期读取全部期刊 → `/news/`(杂志刊头 + 最新一期 + 归档)、`/news/[date]/`(单期页) 、首页速览版块;样式全部复用主站 design token,零新增依赖
- 自动化:定时任务 prompt 即「把这次手动会话蒸馏成的可重复指令」——搜新闻(多角度 5-8 次搜索)→ 写当日 JSON → `npm run build` 验证 → push(push main 自动触发双目标部署,复用既有管线,见 [[内容子项目构建时同步_独立仓库镜像进public_v1]] 同族思路)

**执行过程的两个工程细节:**

- 后台 subagent 并行搜新闻,主线同时写代码,首期内容与板块代码同步就绪,互不阻塞
- 验证时 `preview_screenshot` 连续超时,退级用 DOM 快照 + inspect 逐项核对结构与样式(预览环境验证受限时的降级路径,与 [[Claude预览环境不派发滚动事件_滚动类功能无法行为验证_v1]] 同一族问题)

## 结果分析

- 符合预期:板块风格与主站一致、空数据优雅降级、生产构建一次通过、当日上线
- **未验证部分(诚实边界)**:无人值守运行还没发生过。三个待观察点:① 首跑权限弹窗是否阻塞;② 没有人在环时新闻选稿质量是否下降;③ 期号/日期计算跨日是否正确。本档 v1 先冻结「流程已通、调度已挂」这一层,首跑结果留待补丁

## 方法论沉淀(均为首次发现,暂时性结论)

### ⚠️ 自动化只碰数据律(首次发现)

**核心**:交给定时任务的写权限,收缩到「新增一个数据文件」为止,禁止改任何代码。

**来源**:设计 news.mjs 数据驱动架构时的决策——任务 prompt 里明写「只新增当日 JSON,发现代码问题只报告不修」。

**操作规则**:
1. 呈现层先做成纯数据驱动,数据 schema 写成注释放在读取代码旁边;
2. 自动任务的产出 = 一个新文件,坏了最多缺一期,不会弄坏站点;
3. 构建即校验:JSON 损坏时读取层跳过并告警,不阻断构建。

**反例/边界**:只适用于「增量内容」型自动化;要机器改代码的场景不在此列。

### ⚠️ 人先跑通一遍再交给定时器(首次发现)

**核心**:第一期由人在环完整执行(真实搜集、真实构建、真实 push),再把这次会话蒸馏成定时任务 prompt——prompt 里每一步都被实践校准过,不是凭空设计的。

**来源**:本次先手动产出第 1 期并上线,才创建 `aigc-daily-news` 任务。

**反例/边界**:手动跑通 ≠ 无人值守验证(权限、环境差异仍可能让首跑失败),两层验证要分开记账。

### ⚠️ 宁缺毋滥闸门(首次发现)

**核心**:自动内容生产必须有「不发」的分支——凑不满 5 条真实可溯源新闻就跳过当天,禁止编造;每条必须带真实 URL。

**来源**:任务 prompt 的选稿标准设计。自动化内容的公信力来自它敢于空窗,而不是每天准时有产出。
